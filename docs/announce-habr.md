# Intent-Driven Frontend: выводимые интерфейсы как исследовательская программа

> **Дата:** 2026-04-15
> **Статус:** Черновик анонса для Хабра
> **Жанр:** Обзорная статья с привлечением к спецификации и референсной реализации

## Тезис

Пользовательский интерфейс может быть **выведен**, а не написан. Существует формальная модель, в которой экран, голосовой ответ, вызов к API и документ являются **равноправными материализациями** одного и того же артефакта — пересечения намерений и проекций.

Представленная ниже работа — открытая спецификация этой модели и её JavaScript-референс, проверенные одиннадцатью полевыми тестами суммарного объёма в 572 намерения на девяти доменах различной природы.

---

## Проблема

Современный фронтенд в промышленной разработке описывает один и тот же домен многократно: React-компоненты отдельно, серверные обработчики отдельно, OpenAPI-контракты отдельно, структуры для генерации PDF отдельно. При изменении одного из проявлений остальные рассинхронизируются, и связность поддерживается человеческой дисциплиной, а не структурой.

Попытки унификации — кодогенерация из OpenAPI, схемы GraphQL, meta-компоненты в no-code платформах — устраняют дублирование локально, но не формализуют, **что именно пользователь может сделать** и **что он при этом увидит**. Они остаются либо транспортным контрактом (GraphQL/OpenAPI), либо визуальной разметкой (DSL для no-code). Связь между действием и его последствием фиксируется императивно, в теле обработчика.

Альтернативный подход — сделать **намерение** первоклассной сущностью: задать множество возможных действий декларативно, вывести из них все формы представления, и оставить за исполнителем только причинный каузальный порядок и функцию свёртки. Эта идея восходит к event sourcing и CQRS, но те остаются структурой состояния; IDF расширяет её до структуры **всего взаимодействия** — включая рендеринг.

## Формальное ядро

Центральное уравнение:

```
World(t) = fold(Φ_confirmed)
```

где `Φ` — полный append-only поток подтверждённых эффектов, а `fold` — функция свёртки, применяющая эффекты в порядке, заданном причинным отношением `≺` (топологическая сортировка по `parent_id`). Состояние мира **не хранится** — оно вычисляется из истории.

Каждое намерение `I` определяется пятёркой **частиц**:

- `E` — множество сущностей, на которые оно влияет
- `C` — условия применимости (предикаты над `World(t)`)
- `F` — порождаемые эффекты (α ∈ {`add`, `replace`, `remove`, `batch`})
- `W` — свидетельства (проекции полей мира, известные актору)
- `P` — вид подтверждения (инлайн, форма, модалка, typed-confirmation)

Проекция — пара `⟨E, Q⟩`, где `Q` — запрос над миром, возвращающий данные проекции. Между намерениями и проекциями **не существует прямой связи**: намерение может использоваться из любой проекции, проекция ссылается на сущности и наследует доступные над ними намерения.

Эти определения дают две важные производные:

1. **Алгебра связей** (§12 спеки). Пять типов отношений между намерениями — последовательность `▷`, антагонизм `⇌`, исключение `⊕`, параллельность `∥`, смежность — вычисляются из частиц без декларации автора. Если два намерения производят обратные эффекты на одной сущности, они антагонистичны; если эффекты одного входят в условие другого — последовательны; и так далее.

2. **Кристаллизатор** — функция, принимающая множество намерений, проекций и онтологию, и возвращающая **артефакт**: декларативное описание экранов, маршрутов и элементов управления. На десяти доменах из одиннадцати 78% экранов выведены кристаллизатором механически, без участия автора в вёрстке.

## Четыре равноправные материализации

Артефакт кристаллизатора — чистое промежуточное представление. Над ним построены четыре функции материализации, независимые друг от друга:

| Материализация | Функция | Выход |
|---|---|---|
| **Пиксели** | `renderPixels(artifact, adapter) → React VDOM` | Рендер через один из четырёх UI-адаптеров (Mantine, shadcn/ui, Apple visionOS-glass, Ant Design) |
| **Голос** | `materializeAsVoice(artifact, projection) → { turns }` | Speech-script в трёх форматах: JSON (для voice-агентов), SSML (для TTS), plain (отладка) |
| **Агентский API** | `GET /api/agent/:domain/*` | JWT + role-based access + preapproval guard + typed tool-call; LLM-агент может исполнять намерения напрямую |
| **Документ** | `materializeAsDocument(artifact, projection) → doc-graph` | HTML / JSON-граф для PDF, регуляторной выгрузки, печати |

Формально: четыре функции над одним артефактом. Семантически: домен получает тело приложения, голосового помощника, серверный API и документальную форму — из одного определения, без домен-специфического кода per-materialization.

Равноправие доказано эмпирически: функция `materializeAsDocument` работает для всех четырёх архетипов (`catalog` / `feed` / `detail` / `dashboard`) без per-domain кода и переиспользует ту же `filterWorldForRole`, что и агентский API.

## Верификация через полевые тесты

Спецификация не доказывает свою полноту теоретически. Вместо этого парадигма подвергается серии **полевых тестов** — каждый представляет из себя реальный домен разной природы с полным набором намерений, проекций, ролей и внешних сервисов.

| № | Домен | Намерений | Особенность |
|---|---|---|---|
| 1 | Очередь чтения | ~12 | персональный базовый тест |
| 2 | Календарь событий | ~18 | темпоральный |
| 3 | Интернет-магазин | ~22 | транзакционный |
| 4 | Бронирование услуг | 21 | транз.-темпоральный |
| 5 | Совместное планирование | 17 | коллаборативный |
| 6 | Workflow-редактор | 15 | графовый/инструментальный |
| 7 | Мессенджер | 99 | коммуникационный с WebRTC |
| 8 | Аукционная площадка | 223 | транз.-аукционный, без ManualUI |
| 9 | Дневник эмоций (RULER) | 47 | аналитический + custom canvas |
| 10 | Инвестиции + робо-эдвайзер | 49 | fintech, 4 роли, 3 ML-сервиса |
| 11 | Last-mile food delivery | 45 | операционный, 3 paradigm-additions |

Методология: каждый домен включается до обнаружения **критического блокера** — семантической дыры, которую парадигма не покрывает. Обнаруженный блокер фиксируется как **§26 open item**, формализуется и закрывается в следующем релизе. На одиннадцати доменах критических блокеров не обнаружено; каждый тест либо проходит без расширения ядра, либо порождает **аддитивное** (не ломающее существующие интерфейсы) расширение.

Полевые тесты §8 (meshok-style auction, 223 намерения, 4 роли) и §11 (delivery, три paradigm-additions одновременно) заслуживают отдельного упоминания. Восьмой тест показал, что чисто-кристаллизационный домен (без единой строчки `ManualUI.jsx`) возможен при полном корпусе намерений. Одиннадцатый тест впервые интегрировал три независимо разработанных расширения — темпоральный scheduler первого класса, spatial map-primitive и instance-уровневую необратимость эффектов — в одном домене без коллизий.

Суммарно по девяти доменам прототипа: **572 намерения** на одном движке. Runtime (`fold`, `causalSort`, `intentAlgebra`, `crystallize`) — **не** модифицируется между доменами.

## Честные границы

Спецификация и реализация формально разделены на три категории: «реализовано», «частично реализовано» и «аспирационно». Начиная с версии 1.4, четыре релиза подряд **аспирационная категория пуста**. Это устойчивый признак синхронности манифеста и кода.

Тем не менее, известные ограничения зафиксированы в разделе §26 манифеста:

- **Композитные / полиморфные сущности.** Union-типы в `entity.kind` пока не формализованы. Обход: описание специализаций как отдельных сущностей с общей родительской связкой.
- **Server-rendered PDF / DOCX.** `documentMaterializer` отдаёт HTML и JSON-граф; полноценный PDF через puppeteer + цифровая подпись — production extension.
- **Cluster-friendly single-leader для scheduler.** При горизонтальном масштабе темпоральные таймеры потребуют либо leader-election, либо DB-уровневого lock на transition `firedAt`.
- **Form-factor / device-profile как ось §17.** Решение «разные UI-адаптеры для разных ролей одного домена» (например, kiosk-KDS для мерчанта и mobile-tracker для клиента) требует per-role adapter resolver в рантайме — открытая задача.

Полный перечень: `docs/manifesto-v1.7.md` §23 и §26.

Кроме того, парадигма имплицитно спроектирована под **транзакционные** и **транзакционно-темпоральные** домены. На коммуникационных доменах (мессенджер) парадигма работает, но нагрузка на витнессы WebRTC-состояния выходит за пределы комфорта. На чисто-графовых инструментах (визуальные редакторы процессов) ядро обеспечивает CRUD, но не автоматическое исполнение. Этот уклон зафиксирован в §21 манифеста как «честная граница».

## Открытая спецификация

Формальная часть работы опубликована под лицензией **Apache License 2.0**:

> **Intent-Driven Framework — Specification v1.0, Part 1: Core Model**
> <https://github.com/DubovskiyIM/idf/tree/main/spec>

Состав:

- `idf-v1.0-part1-core.md` — нормативный документ (~920 строк, RFC 2119) с формальными определениями онтологии, намерений, эффектов, алгебры, интегритета
- `conformance/` — 54 языконезависимых JSON-теста в трёх уровнях соответствия (L1 Core: 24, L2 Algebra: 15, L3 Full: 15)
- `schemas/` — пять JSON Schema Draft-07 для `intent`, `effect`, `ontology`, `world`, `domain`

Любой может реализовать спецификацию на любом языке, под любой лицензией, для любой цели. Тесты — один JSON-файл на проверку; минимальный runner занимает ~30 строк на JavaScript, Python или Go (примеры в `spec/conformance/README.md`).

## Референсная реализация

Реализация для JavaScript распространяется через npm под scope `@intent-driven`:

```bash
npm install @intent-driven/core @intent-driven/renderer @intent-driven/adapter-mantine
```

- `@intent-driven/core@0.3.0` — ядро: fold, алгебра, кристаллизатор, четыре материализатора, инварианты, фильтрация по ролям. **Business Source License 1.1** с Change Date 2030-04-15 (автопереход на Apache 2.0 через 4 года). Некоммерческое и внутреннее производственное использование свободно; managed hosting — требует коммерческой лицензии.
- `@intent-driven/renderer@0.2.0` — рендерер: 7 архетипов, примитивы (включая spatial `map` v1.7 и `IrreversibleBadge`), реестр адаптеров. **MIT.**
- `@intent-driven/adapter-{mantine,shadcn,apple,antd}@0.1.1` — четыре UI-адаптера, различающиеся эстетикой. **MIT.**
- `@intent-driven/canvas-kit@0.1.0` — SVG-утилиты для custom canvas, standalone. **MIT.**

Референс проходит все 54 conformance-теста на уровне L3.

## Демонстрация

Работающий прототип показывает переключение между девятью доменами без перезагрузки рантайма. Каждый домен представлен через один из четырёх визуальных адаптеров. Переключение UI-kit'а также динамическое.

Код прототипа — `src/domains/*/` и `src/prototype.jsx` в <https://github.com/DubovskiyIM/idf>. Запуск локально:

```bash
git clone https://github.com/DubovskiyIM/idf.git && cd idf
npm install
npm run server    # :3001
npm run dev       # :5173 — переключатель доменов
```

Для проверки материализаций:

```bash
curl http://localhost:3001/api/agent/booking/schema
curl http://localhost:3001/api/voice/reflect/home?format=ssml
curl http://localhost:3001/api/document/invest/portfolio?format=html
```

Делий `delivery` демонстрирует одновременную работу трёх paradigm-additions: темпоральный scheduler порождает авто-отмену заказа при отсутствии подтверждения; map-primitive рендерит карту диспетчера с полигонами зон и маркерами курьеров; capture_payment помечает эффект как необратимый, блокируя `α:"remove"` на уровне валидатора.

Landing-страница с интерактивными визуализациями архетипов и токеномикой кристаллизации: `landing/index-ru.html` (раздел футера ссылается на GitHub).

## Область применения

Парадигма не является серебряной пулей. Она применима в доменах, где:

1. Множество действий конечно и формально описуемо (не игры реального времени, не низкоуровневая графика).
2. Состояние — дискретное и пошаговое (не потоковая обработка медиа).
3. Несколько материализаций одного домена имеют бизнес-ценность (операционное приложение + голосовой ассистент + выгрузка для регулятора + API для партнёров).

Для одиночного мобильного приложения или чисто-маркетингового лендинга накладные расходы парадигмы не оправдываются. Для многостороннего операционного приложения или платформы со сторонними интеграциями — парадигма позволяет выйти из режима «писать каждую материализацию отдельно» в режим «описывать домен один раз».

Токеномика кристаллизации: первичный рост — ~10–12K токенов для 14 намерений; полный домен (52 намерения) — ~50–60K токенов. Инкрементальная пере-кристаллизация (при добавлении одного намерения) — ~80 мс, <1K токенов. LLM участвует в **design-time**, не в runtime; runtime — чистые функции.

## Приглашение

Спецификация и реализация открыты. Конструктивная критика приветствуется:

- Найденная семантическая дыра в спецификации, не покрытая текущими 54 conformance-тестами, — issue в <https://github.com/DubovskiyIM/idf/issues>
- Альтернативная реализация на Rust / Python / Go / Kotlin — ссылка на репозиторий со ссылкой на conformance-тесты, проходящие в вашей имплементации
- Домен, который парадигма, по вашей оценке, не выдержит, — комментарий с описанием (полевой тест 12 ждёт кандидата)
- Расширение под специфику вашего стека — pull request или дизайн-документ

Для тех, кому интересна внутренняя механика — одиннадцать отчётов о полевых тестах (`docs/field-test-*.md`) документируют каждый цикл «обнаружение блокера → формализация → закрытие». Манифест v1.7 (`docs/manifesto-v1.7.md`) — 26 разделов с техническими решениями.

Обсуждение в комментариях.
