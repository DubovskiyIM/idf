# 0. Abstract, Status, Conformance Language

## 0.1 Abstract

**Intent-Driven Frontend (IDF)** — это формат описания приложения, основанный на декларации пользовательских *намерений* и *проекций*. Приложение, описанное в формате IDF, пригодно для автоматического производства четырёх равноправных материализаций:

- **pixels** — UI для человека (React / SwiftUI / любой другой UI-стек через адаптер);
- **voice** — голосовой диалог (turn-structured поток для TTS / voice-agent);
- **agent-API** — REST / RPC-интерфейс для LLM-агентов;
- **document** — статичный HTML / PDF / JSON для чтения и архива.

Формат состоит из четырёх взаимосвязанных структур: **онтологии** (сущности, роли, инварианты, правила), **намерений** (атомы возможных действий), **проекций** (читающие структуры) и **артефакта v2** (детерминированный выход кристаллизации). Первые три — *вход* системы (создаётся автором или LLM); артефакт v2 — *выход* (потребляется реализациями).

Формат не предписывает язык программирования, UI-библиотеку, СУБД, транспорт и стиль визуального дизайна. Формат предписывает только форму данных и семантику их композиции.

IDF конкурирует не с UI-фреймворками, а с форматами описания приложений / API — такими как OpenAPI / AsyncAPI / JSON-LD. В отличие от них, IDF покрывает одновременно модель домена, доступные действия и производные поверхности UI / голоса / агента.

## 0.2 Status of this Document

Это **Working Draft** спецификации IDF версии 0.1. Документ отражает состояние формата, зафиксированное в reference-реализации `@intent-driven/core@0.7.0` и прототипе `idf` версии v1.9 (2026-04-16).

Документ **не является стабильным**: секции `draft` могут измениться несовместимым образом до перехода в статус `Candidate Recommendation`. Секции `planned` зарезервированы, но ещё не написаны.

Жизненный цикл спецификации:

1. **Working Draft (WD)** — текущий статус v0.1. Формат эволюционирует вместе с reference-реализацией.
2. **Candidate Recommendation (CR)** — заморозка на несколько месяцев для сбора обратной связи от независимых имплементаторов. Минимум **два независимых conformant implementation** (помимо reference) необходимы для перехода в CR.
3. **Recommendation (REC)** — стабильная версия формата. Следующие изменения требуют новой major-версии спецификации.

## 0.3 Conformance language

В нормативных секциях документа ключевые слова **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, **OPTIONAL** интерпретируются согласно [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) и [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174), но **только** когда они выделены **жирным** шрифтом.

Обычное (не-жирное) использование этих слов в тексте не имеет нормативного значения.

Пример:

> Артефакт **MUST** содержать поле `version` со значением `2`.

— нормативное требование.

> В типичном случае авторы используют английский язык для полей `label`, но русский тоже работает.

— ненормативное описание.

## 0.4 Отношения с другими документами

- **IDF Manifesto** — философский / исторический документ. Ненормативный. Манифест описывает *мотивацию* парадигмы и её *полевые тесты*. Расхождения между манифестом и спецификацией разрешаются в пользу спецификации.
- **Reference Implementations** — конкретные пакеты, реализующие формат. Reference-реализация авторитетна для уточнения *неоднозначности формата*, но не замещает спецификацию: поведение, не описанное в спецификации, — не часть формата, даже если оно есть в reference-коде.
- **Conformance Test Suite** (`tests/conformance/`) — корпус fixture'ов + ожидаемых выходов. При расхождении формата и тест-сьюта авторитетна спецификация; тест-сьют подлежит исправлению.

## 0.5 Терминология

| Термин | Определение |
|---|---|
| **Артефакт v2** | JSON-объект фиксированной формы, описывающий одну проекцию приложения. Выход `crystallizeV2()`. |
| **Онтология** | JSON-объект, описывающий сущности, роли, правила и инварианты домена. Вход `crystallizeV2()`. |
| **Намерение (intent)** | Атомарная декларация возможного действия пользователя или агента, с частицами `⟨E, C, F, W, P⟩`. |
| **Проекция (projection)** | Первоклассная читающая структура. Input кристаллизатора; после кристаллизации превращается в артефакт v2. |
| **Кристаллизация** | Детерминированная функция `(INTENTS, PROJECTIONS, ONTOLOGY) → {projectionId: Artifact}`. |
| **Архетип** | Одно из семи значений `feed / catalog / detail / form / canvas / dashboard / wizard`, задающее скелет артефакта. |
| **Слот** | Именованная позиция в артефакте (`body`, `toolbar`, `overlay`, ...), куда попадают control'ы. |
| **Control-архетип** | Один из ~10 способов материализовать намерение в UI-контрол (`composerEntry`, `formModal`, `confirmDialog`, ...). |
| **Материализация** | Трансформация артефакта v2 в конкретный выход (pixels / voice / agent-API / document). |
| **Conformant implementation** | Программа / библиотека, которая корректно парсит формат IDF и производит одну из материализаций с соблюдением normative требований спецификации. |
| **Conformance class (L1–L4)** | Уровень поддержки формата реализацией. L1 — минимальное ядро; L4 — полный набор включая capability surface. Подробности в §3. |

## 0.6 Версионирование

Спецификация следует SemVer:

- **Major** (`1.0` → `2.0`) — несовместимое изменение формата. Старые артефакты не валидны в новой major-версии.
- **Minor** (`0.1` → `0.2`) — additive изменения. Старые артефакты валидны. Новые реализации MAY отказаться рендерить артефакты без новых полей.
- **Patch** (`0.1.0` → `0.1.1`) — уточнения формулировок, исправление опечаток, не затрагивают форму данных.

Артефакт v2 декларирует `version: 2`. Это — major-версия **формата артефакта**, отдельная от версии спецификации. Формат артефакта менялся: `v1` (устаревший, не покрывается данной спецификацией) → `v2` (текущий). Спецификация v0.x фиксирует артефакт v2.
