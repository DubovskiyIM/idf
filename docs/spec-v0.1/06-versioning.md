# 6. Versioning and Backward Compatibility

Эта глава определяет версионирование спецификации IDF, правила backward compatibility и процесс внесения изменений.

## 6.1 Два независимых типа версий

Две величины версионируются отдельно:

| Что | Пример | Область |
|---|---|---|
| **Версия спецификации** | `v0.1`, `v0.2`, `v1.0`, `v2.0` | Документ целиком: текст + все JSON Schema'ы + test suite |
| **Версия формата артефакта** | `version: 2` | Поле внутри артефакта. Major-версия **формата данных**. |

Версия спецификации следует SemVer. Версия артефакта — монотонно растущий integer, меняется только при несовместимых структурных изменениях формата данных.

На момент v0.1 спецификации артефакт — версии **2** (v1 устарел и не покрывается).

## 6.2 SemVer спецификации

### Major (`0.x` → `1.0`, `1.x` → `2.0`)

**Несовместимое** изменение нормативных требований:

- удалены / переименованы обязательные поля в schema;
- изменена семантика `fold` / `composition` / `anchoring`;
- изменён enum с удалением значений;
- ужесточены `MUST`-правила, ранее бывшие `MAY`.

Major-bump требует обновления conformant implementations. Старые артефакты **MAY** перестать быть валидными.

### Minor (`0.1` → `0.2`, `1.0` → `1.1`)

**Additive** изменения:

- добавлены новые опциональные поля в schema;
- добавлены новые значения в enum;
- добавлены новые control-архетипы / primitive-категории / invariant-kinds;
- добавлены новые материализационные track'и;
- добавлены новые conformance classes (L5, L6 ...);
- ужесточены `MAY`-правила до `SHOULD`, но не до `MUST`.

Minor-bump — backward-compatible на уровне данных: старые артефакты **MUST** оставаться валидными. Но старые реализации **MAY** не поддерживать новые features — они декларируют это через `conformance.json`.

### Patch (`0.1.0` → `0.1.1`)

**Уточнение формулировок, не меняющее формат данных:**

- исправления опечаток;
- разъяснения неоднозначных требований (без фактического изменения ожидаемого поведения);
- добавление / исправление примеров;
- дополнения к терминологии.

Patch-bump не требует обновления реализаций. Прогон того же теста-сьюта должен давать тот же результат.

## 6.3 SemVer формата артефакта

Поле `version` артефакта ≠ версия спецификации. `version: 2` означает major-формат. Переход `version: 2 → version: 3` — редкое событие, происходит **только** при несовместимом структурном изменении формы артефакта (не добавлении полей, а реструктуризации).

Ожидаемая траектория: `v1` (устарел) → `v2` (текущий, стабилизирован в спецификации v0.1) → `v3` (гипотетически, при необходимости в далёком будущем).

Спецификация v1.0 **MAY** продолжать описывать артефакт `v2`. Спецификация v2.0 **MAY** описывать `v2` или перейти на `v3`.

## 6.4 Минимальные артефакты совместимости

### 6.4.1 Forward compatibility — парсер `v0.1`, артефакт `v0.2`

Parser, заявляющий спецификацию v0.1, **MUST** корректно обрабатывать артефакты, сгенерированные по v0.2, при условии, что они не используют новых required-полей. Механизм:

- Неизвестные поля top-level → игнорировать без падения (§3.9.4).
- Неизвестные значения enum в `archetype`, `alpha`, `control`, `type` → вести себя согласно §3.9.
- Неизвестные invariant-kinds → warning + continue (§3.9.5).

### 6.4.2 Backward compatibility — парсер `v0.2`, артефакт `v0.1`

Parser v0.2 **MUST** принимать артефакты, сгенерированные по v0.1, без изменений. Это базовое требование backward compat.

Исключение — только major-bump.

## 6.5 Deprecation policy

Отказ от опции / поля / правила проходит три фазы:

1. **Deprecated** — в следующем minor-релизе поле остаётся рабочим, но помечается `"deprecated": true` в schema + упоминание в CHANGELOG. Реализации **MUST** продолжать его поддерживать.
2. **Discouraged** — через 2+ minor-релиза новое использование **SHOULD NOT** добавляться, но старое продолжает работать. Deprecated-warning повышается до `SHOULD`.
3. **Removed** — только в major-релизе. Поле удаляется из schema.

Минимальный срок между Deprecated и Removed: **один major cycle**.

## 6.6 Эволюция enum'ов

Добавление значения в enum — **minor**. Удаление / переименование — **major**.

Нормативное требование (§3.9): реализация **MUST** уметь обрабатывать неизвестные enum-значения без падения. Это делает добавление enum-значения формально additive, хотя фактически реализация без обновления не поддерживает новое значение.

Пример: добавление `archetype: "timeline"` в v0.2 — minor. Реализации v0.1 увидят артефакт с неизвестным архетипом и либо отклонят с `unsupported archetype: timeline`, либо game-gracefully пропустят. Никакое падение не допустимо.

## 6.7 Процесс внесения изменений

1. **Предложение** — issue / discussion в репозитории спецификации, с описанием:
   - что меняется;
   - почему (use case, field-test evidence, согласованность);
   - уровень изменения (patch / minor / major);
   - impact на reference-реализации;
   - тест-фикстуры для добавления / изменения.

2. **Обсуждение** — минимум **14 дней** открытой дискуссии для minor/major. Patch — 3 дня.

3. **Prototype** — изменение реализуется в reference-implementations как draft PR.

4. **Ratification** — если две независимые L2+ реализации подтверждают изменение (или пост v1.0 — спецификация в статусе REC), изменение принимается.

5. **Release** — обновление CHANGELOG, bump версии, публикация нового релиза.

До v1.0 (Working Draft) процесс упрощён — ratification не требуется, автор спецификации вправе вносить изменения с 3-дневным уведомлением.

## 6.8 CHANGELOG

Файл `CHANGELOG.md` (planned) в директории спецификации **MUST** перечислять все изменения по релизам в формате Keep a Changelog (https://keepachangelog.com).
