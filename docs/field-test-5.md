# Intent-Driven Frontend — Полевой тест 0.5.1

## Проверка парадигмы на коллаборативном домене

## Домен: совместное планирование встреч (Doodle-подобное)

Сервис для планирования встреч в группе: организатор создаёт опрос с вариантами времени, участники голосуют за доступные слоты, система находит оптимальный вариант, организатор подтверждает, встреча создаётся. Поддержка: повторные опросы, предложение альтернатив участниками, отклонение приглашений, отмена.

**Почему этот домен.** Манифест (раздел 21) прямо указывает: коллаборативные домены — зона наименьшей силы парадигмы. Четыре предыдущих теста показали конвергенцию блокеров (4→4→2→0), но все на одно/малоакторных доменах. Совместное планирование — **минимальный коллаборативный домен**: несколько акторов, переговорный процесс, распределённый коммит. Если парадигма выживает здесь — она выживает за пределами транзакционной зоны. Если нет — мы честно фиксируем границу.

Ключевые стресс-точки:
- **Многоакторная координация**: одно действие (планирование встречи) распределено между организатором и N участниками
- **Переговорные намерения**: пространство возможностей (варианты времени) → голосование → сужение → коммит
- **Рабочие процессы**: poll_created → voting → closed → resolved → meeting_created — разные акторы на разных фазах
- **Распределённый коммит**: организатор коммитит результат, но результат зависит от голосов участников

## Онтология

**Сущности:**

`Organizer` — внутренняя. Поля: id, name, email. Создатель опроса.

`Participant` — внутренняя. Поля: id, name, email, status {invited, active, declined}. Приглашённый участник.

`Poll` — внутренняя. Поля: id, organizerId, title, description, status {draft, open, closed, resolved, cancelled}, createdAt. Жизненный цикл: draft → open → closed → resolved | cancelled.

`TimeOption` — внутренняя. Поля: id, pollId, date, startTime, endTime, votes_yes, votes_no, votes_maybe. Вариант времени в опросе.

`Vote` — внутренняя. Поля: id, participantId, optionId, pollId, value {yes, no, maybe}. Голос участника.

`Meeting` — внутренняя. Поля: id, pollId, organizerId, title, date, startTime, endTime, participantIds, status {confirmed, cancelled}. Создаётся при resolve_poll.

**Предикаты:**
- `poll_is_open(poll)` ≡ `poll.status = 'open'`
- `poll_is_closed(poll)` ≡ `poll.status = 'closed'`
- `participant_is_active(participant)` ≡ `participant.status = 'active'`
- `has_votes(poll)` ≡ `∃ vote: vote.pollId = poll.id`
- `best_option(poll)` ≡ `option с max(votes_yes) и min(votes_no)`

**Правила импликации:**
- Закрытие опроса → можно разрешить (resolve)
- Разрешение → создаётся встреча
- Отмена опроса → все связанные голоса неактуальны
- Отклонение приглашения → голоса участника удаляются

## 20 намерений

### Организатор (9)

**1. create_poll — Создать опрос**
```
I = ⟨E, C, F, W, P⟩
E: poll: Poll
C: []
F: [{ α: add, target: polls, σ: account }]
W: []
P: click
creates: Poll(draft)
```
Создаёт опрос в статусе `draft`. Организатор добавляет варианты времени и участников перед открытием.

**2. add_time_option — Добавить вариант времени**
```
E: poll: Poll, option: TimeOption
C: [poll.status IN ('draft', 'open')]
F: [{ α: add, target: options, σ: account }]
W: [poll.title, existing_options.count]
P: click
creates: TimeOption
```

**3. remove_time_option — Удалить вариант**
```
E: option: TimeOption
C: [poll.status IN ('draft', 'open')]
F: [{ α: remove, target: options, σ: account }]
W: [option.date, option.startTime, option.votes_yes]
P: click (подтверждение если есть голоса)
irreversibility: medium (если есть голоса — данные теряются)
```

**4. invite_participant — Пригласить участника**
```
E: poll: Poll, participant: Participant
C: [poll.status IN ('draft', 'open')]
F: [{ α: add, target: participants, σ: account }]
W: [poll.title, participants.count]
P: click
creates: Participant(invited)
```
Сигнал: `invitation_sent` (email/push участнику).

**5. open_poll — Открыть опрос для голосования**
```
E: poll: Poll
C: [poll.status = 'draft', poll.options.count > 0, poll.participants.count > 0]
F: [{ α: replace, target: poll.status, value: 'open', σ: account }]
W: [options.count, participants.count]
P: click
```
Последовательная связь: `create_poll ▷ add_time_option ▷ invite_participant ▷ open_poll`. Сигнал: `poll_opened` (уведомление всем участникам). Точка, после которой участники могут голосовать.

**6. close_poll — Закрыть голосование**
```
E: poll: Poll
C: [poll.status = 'open']
F: [{ α: replace, target: poll.status, value: 'closed', σ: account }]
W: [votes.count, participation_rate, best_option.summary]
P: click
```
Антагонист для голосования: после close новые голоса невозможны.

**7. resolve_poll — Выбрать время и создать встречу**
```
E: poll: Poll, winning_option: TimeOption
C: [poll.status = 'closed']
F: [{ α: replace, target: poll.status, value: 'resolved', σ: account },
    { α: add, target: meetings, σ: account }]
W: [winning_option.date, winning_option.startTime, votes_yes.count, votes_no.count, conflicting_participants]
P: click (выбор варианта)
creates: Meeting
```
Ключевое переговорное намерение: организатор видит результаты голосования и выбирает вариант. Система предлагает лучший (`best_option`), но организатор может выбрать другой. Сигнал: `meeting_confirmed` (уведомление всем).

**8. cancel_poll — Отменить опрос**
```
E: poll: Poll
C: [poll.status IN ('draft', 'open', 'closed')]
F: [{ α: replace, target: poll.status, value: 'cancelled', σ: account }]
W: [poll.title, participants.count, votes.count]
P: click (подтверждение)
irreversibility: high
```
Сигнал: `poll_cancelled` (уведомление если были участники).

**9. cancel_meeting — Отменить встречу**
```
E: meeting: Meeting
C: [meeting.status = 'confirmed']
F: [{ α: replace, target: meeting.status, value: 'cancelled', σ: account }]
W: [meeting.title, meeting.date, participants.count]
P: click (подтверждение)
irreversibility: high
antagonist: resolve_poll (частичный — отменяет meeting, но не poll)
```
Сигнал: `meeting_cancelled` (уведомление всем участникам).

### Участник (7)

**10. accept_invitation — Принять приглашение**
```
E: participant: Participant
C: [participant.status = 'invited']
F: [{ α: replace, target: participant.status, value: 'active', σ: account }]
W: [poll.title, poll.organizer.name, options.count]
P: click
```
Последовательная: `invite_participant ▷ accept_invitation ▷ vote_*`.

**11. decline_invitation — Отклонить приглашение**
```
E: participant: Participant
C: [participant.status IN ('invited', 'active')]
F: [{ α: replace, target: participant.status, value: 'declined', σ: account }]

antagonist: accept_invitation
```
Сигнал: `participant_declined` (уведомление организатору).

**12. vote_yes — Голосовать «доступен»**
```
E: vote: Vote, option: TimeOption
C: [poll_is_open(poll), participant_is_active(participant)]
F: [{ α: add, target: votes, σ: account, value: 'yes' }]
W: [option.date, option.startTime, current_votes_summary]
P: click
creates: Vote(yes)
```

**13. vote_no — Голосовать «недоступен»**
```
E: vote: Vote, option: TimeOption
C: [poll_is_open(poll), participant_is_active(participant)]
F: [{ α: add, target: votes, σ: account, value: 'no' }]
W: [option.date, option.startTime]
P: click
creates: Vote(no)
```

**14. vote_maybe — Голосовать «возможно»**
```
E: vote: Vote, option: TimeOption
C: [poll_is_open(poll), participant_is_active(participant)]
F: [{ α: add, target: votes, σ: account, value: 'maybe' }]
W: [option.date, option.startTime]
P: click
creates: Vote(maybe)
```

**15. change_vote — Изменить голос**
```
E: vote: Vote
C: [poll_is_open(poll), participant_is_active(participant)]
F: [{ α: replace, target: vote.value, σ: account }]
W: [option.date, vote.value (текущий)]
P: click
```
Антагонист самого себя в каком-то смысле — перезаписывает предыдущий голос.

**16. suggest_alternative — Предложить альтернативное время**
```
E: poll: Poll, option: TimeOption
C: [poll_is_open(poll), participant_is_active(participant)]
F: [{ α: add, target: options, σ: account }]
W: [poll.title, existing_options]
P: click
creates: TimeOption (с маркером suggested_by: participant.id)
```
Участник предлагает вариант, которого нет в опросе. Организатор видит предложение. Остальные участники могут голосовать за него.

### Граничные / системные (4)

**17. sync_participant_calendar — Проверить доступность участника**
```
E: participant: Participant
C: [participant.calendarConnected = true]
F: [автоматические голоса на основе свободно/занято]
W: [available_slots, busy_slots]
P: auto
```
Граница: если участник подключил Google Calendar, система автоматически голосует за доступные слоты. Зеркальная сущность: календарь участника.

**18. auto_suggest_best — Предложить лучшее время**
```
E: poll: Poll
C: [poll.status = 'closed', has_votes(poll)]
F: [] (только свидетельства, без эффектов — это проекция-рекомендация)
W: [best_option, participation_rate, conflicts]
P: нет (автоматическое)
```
Система анализирует голоса и рекомендует лучший вариант. Это не намерение в строгом смысле — это проекция (нет эффектов). Но она информирует `resolve_poll`.

**19. set_deadline — Установить дедлайн голосования**
```
E: poll: Poll
C: [poll.status = 'open']
F: [{ α: replace, target: poll.deadline, σ: account }]
W: [poll.current_deadline, votes.count, participation_rate]
P: click
```
Запланированный сигнал: при наступлении дедлайна → `deadline_reached` → организатор получает уведомление.

**20. send_reminder — Отправить напоминание**
```
E: poll: Poll
C: [poll.status = 'open', ∃ participant: participant.votes.count = 0]
F: [] (только сигнал)
W: [participants_without_votes]
P: click
```
Сигнал: `voting_reminder` (тем участникам, кто ещё не голосовал).

## 4 проекции

**poll_overview** `V = ⟨E, Q, W⟩`
```
E: Poll, TimeOption, Vote, Participant
Q: один опрос со всеми деталями
W: [title, status, options с голосами, participants со статусами, participation_rate]
```
Зависимость от зрителя: организатор видит все голоса поимённо. Участник видит только свои голоса и агрегат (3 да / 1 нет / 2 возможно).

**my_polls** `V = ⟨E, Q, W⟩`
```
E: Poll, Meeting
Q: все опросы, где я организатор или участник
W: [title, status, my_role, next_action_needed]
```

**voting_matrix** `V = ⟨E, Q, W⟩`
```
E: TimeOption, Vote, Participant
Q: матрица голосов: строки = участники, столбцы = варианты, ячейки = yes/no/maybe
W: [participant.name, option.date, option.startTime, vote.value]
```
Классическая Doodle-матрица. Зависимость от зрителя: организатор видит всех, участник видит только себя (или всех — настройка опроса).

**meeting_details** `V = ⟨E, Q, W⟩`
```
E: Meeting, Participant
Q: подтверждённая встреча со всеми деталями
W: [title, date, startTime, endTime, participants, organizer]
```

## Что работает с ходу

**Базовый флоу как цепочка намерений.** create_poll → add_time_option → invite_participant → open_poll → [голосование] → close_poll → resolve_poll → meeting создана. Каждый шаг — намерение с условиями, эффектами, свидетельствами. Последовательные связи выводятся из частиц. Работает.

**Голосование как эффекты.** Каждый голос — эффект `add` в поток Φ. Подсчёт — fold. Изменение голоса — `replace`. Алгебра корректна: голоса разных участников на один вариант — параллельные эффекты (`add` + `add` = объединение). Работает.

**Сигналы.** Уведомления при приглашении, открытии, подтверждении — fire-and-forget при коммите. Работает.

**Проекции.** Матрица голосов, обзор опроса, список моих опросов — чистые проекции. Работает.

**Антагонисты.** accept ⇌ decline, open ↔ cancel. Автовывод из частиц. Работает.

## Находки

### D1 (🔴 блокирующая). Рабочий процесс как первоклассный концепт.

Жизненный цикл опроса `draft → open → closed → resolved` — это не просто цепочка намерений. Это **рабочий процесс**: гетерогенная последовательность фаз, где на каждой фазе разные акторы имеют разные доступные намерения.

- `draft`: только организатор (add_option, invite, open_poll)
- `open`: организатор (add_option, close_poll, cancel) + участники (vote, change_vote, suggest)
- `closed`: только организатор (resolve_poll, cancel)
- `resolved`: только организатор (cancel_meeting)

Наша модель условий (`poll.status = 'open'`) технически покрывает это — намерение голосования неприменимо если poll не open. Но это **имплицитный** рабочий процесс: фазы не описаны явно, доступные намерения на каждой фазе не документированы, переходы между фазами не верифицируются.

Проблема: если автор описывает 20 намерений, и у каждого условие `poll.status = X`, **модель не знает, что это фазы одного процесса**. Она видит 20 независимых намерений с условиями. Связь `close_poll ▷ resolve_poll` выводится, но концепт «фаза» — нет.

Это третий домен с этой находкой (тест 2: планирование встречи, тест 4: шумный дрейф как микро-процесс, этот тест: полный рабочий процесс). **Рекомендация: стабилизировать рабочие процессы.**

Решение-эскиз: **процесс** = `⟨фазы, переходы, роли⟩`. Фаза = предикат на сущности. Переход = намерение, меняющее фазу. Роль = подмножество акторов, имеющих доступ к намерениям на данной фазе. Явная декларация процесса при кристаллизации позволяет: (1) визуализировать процесс как граф фаз, (2) верифицировать полноту (нет тупиков, каждая фаза достижима), (3) генерировать UI по фазам.

### D2 (🔴 блокирующая). Многоакторная координация — распределённый коммит.

`resolve_poll` — это коммит, который зависит от действий N других акторов (голосов). Организатор не может коммитить пока нет голосов. Но он не контролирует, когда участники голосуют.

В нашей модели `resolve_poll` имеет условие `poll.status = 'closed'`. Организатор закрывает голосование, смотрит результаты, выбирает вариант. Технически работает: resolve — это обычное намерение организатора. Но **семантически** это распределённый коммит: результат зависит от действий людей, которых организатор не контролирует.

Проблемы:
1. **Кворум.** Когда закрывать? Если 2 из 5 проголосовали — достаточно? Нужна политика кворума: `close_when: all_voted | quorum(N) | deadline | manual`.
2. **Дедлайн.** Организатор устанавливает deadline → запланированный сигнал → автоматическое закрытие или напоминание. Запланированные сигналы уже стабилизированы — хорошо.
3. **Неголосовавшие.** Как считать неголосовавших? Как abstain? Как no? Нужна политика: `absent_vote: abstain | no | exclude`.

Это второй домен с многоакторной координацией (тест 2: RSVP на встречу, этот тест: голосование). Но здесь проблема острее: в тесте 2 RSVP было бинарным (да/нет), здесь — переговорный процесс с множеством вариантов.

### D3 (🟡 значительная). Переговорные намерения — открытие/сужение/коммит.

Флоу голосования — это не просто CRUD. Это **переговорный процесс**:
1. **Открытие** — организатор предлагает варианты (пространство возможностей)
2. **Расширение** — участники предлагают альтернативы (suggest_alternative)
3. **Сужение** — голосование сужает пространство
4. **Коммит** — организатор выбирает из сужённого пространства

Наша модель описывает каждый шаг как отдельное намерение. Но переговорный паттерн как целое — не описывается. Модель не знает, что `suggest_alternative`, `vote_yes`, `vote_no` — это фазы одного переговорного процесса, а не независимые действия.

Связано с D1 (рабочие процессы) и D2 (многоакторная координация). Возможно, переговорные намерения — это частный случай рабочего процесса с ролью «переговорщик». Второй домен (тест 2: голосование за время встречи).

### D4 (🟡 значительная). Агрегированные свидетельства через голоса.

`resolve_poll` нуждается в свидетельстве `best_option` — вариант с максимумом `yes` и минимумом `no`. Это не поле сущности, а **агрегированная проекция** через голоса.

В нашей модели это проекция `auto_suggest_best`. Но свидетельства намерения — это «проекции, ограниченные контекстом одного рассматриваемого намерения» (раздел 5 манифеста). Значит, свидетельство `best_option` для `resolve_poll` = проекция `voting_matrix`, свёрнутая до рекомендации.

Работает без нового концепта — агрегация через проекцию, привязанную к свидетельству. Но показывает, что **свидетельства могут быть нетривиально вычисляемыми** — не просто «покажи поле», а «агрегируй N голосов, примени политику, выдай рекомендацию».

### D5 (🟡 значительная). Зависимость от зрителя — критична для приватности.

`voting_matrix` для организатора: все голоса поимённо. Для участника: только свои + агрегат. Это не декоративная фича — это **требование приватности**. Если участник видит голоса других, это влияет на честность голосования (стадный эффект).

Зависимость от зрителя уже стабилизирована (раздел 5). Но здесь она не просто меняет содержимое — она меняет **поведение**. Приватное голосование vs публичное — это доменное решение автора, и оно должно быть выражено в определении проекции.

### D6 (🟢 минорная). Сущности с создателем-не-автором.

`suggest_alternative` создаёт TimeOption, но её автор — участник, не организатор. В предыдущих доменах все сущности создавались одним актором. Здесь — совместное создание.

Модель не ломается: `suggest_alternative` — намерение участника, создающее эффект `add` в `options`. Организатор видит его как обычный вариант (с маркером `suggested_by`). Но кристаллизация должна отличать «варианты организатора» от «предложений участников» визуально.

| D6 | Сущности с создателем-не-автором | 🟢 | 5 (новое) |

Два блокера. Три значительных. Одна минорная.

---

## Что было подтверждено

- **Базовый флоу** — цепочка намерений с условиями и связями покрывает CRUD-часть опроса.
- **Голосование как эффекты** — `add` в поток Φ, подсчёт через fold. Алгебра корректна.
- **Запланированные сигналы** — дедлайн голосования = запланированный сигнал. Стабильный концепт подтверждён ещё раз.
- **Зависимость от зрителя** — необходима и работает. Подтверждена третий раз.
- **Антагонисты** — accept ⇌ decline. Автовывод корректен.
- **Проекции** — матрица голосов, обзор, список — чистые проекции.

---

## Что парадигма НЕ покрывает

Честная фиксация:

1. **Фазы процесса не первоклассные.** Модель выражает фазы через условия на статусе, но не знает, что это фазы одного процесса. Это означает: кристаллизация не может сгенерировать UI «по фазам» (показывай разные экраны для draft/open/closed), потому что не видит процесс.

2. **Кворум и политики голосования.** Модель не имеет концепта «достаточно голосов для перехода». Это частный случай многоакторной координации. Решение: расширить условия намерений предикатами вида `quorum_reached(poll, 0.8)`.

3. **Переговорный паттерн как целое.** Открытие пространства → голосование → сужение → коммит — это высокоуровневый паттерн, который модель не видит. Каждый шаг — отдельное намерение. Возможно, это не проблема: парадигма оперирует на уровне намерений, а паттерн — метауровень.

## Рекомендации для манифеста v0.6

1. **Стабилизировать рабочие процессы** (тесты 2, 4, 5). Ввести процесс как `⟨фазы, переходы, роли⟩`. Это расширяет онтологию, не меняет ядро.

2. **Не стабилизировать многоакторную координацию** — ещё 2 домена. Зафиксировать как открытое расширение с эскизом решения (кворум, политики голосования).

3. **Переговорные намерения** — возможно, частный случай рабочих процессов. Проверить: если рабочие процессы стабилизированы, покрывают ли они переговорный паттерн?

## Вердикт

Коллаборативный домен обнаружил два блокера, оба предсказанных. Парадигма работает на уровне отдельных намерений и эффектов, но **не видит процесс как целое**. Рабочие процессы — зрелый кандидат на стабилизацию (3 домена). Многоакторная координация — открытая проблема.

Ядро парадигмы (эффекты, причинность, алгебра, проекции, сигналы) прошло пятый тест без ревизии. Блокеры — на уровне расширений, не ядра. Парадигма фальсифицируема в коллаборативной зоне, но не опровергнута.
