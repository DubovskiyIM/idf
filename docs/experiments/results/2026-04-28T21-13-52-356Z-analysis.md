## Главный вывод

**Meta-circular compounding наблюдаем эмпирически.**

> При work item'е, который структурно требует self-modification, LLM создаёт новый intent через `propose_meta_intent`. На последующих work item'ах того же класса LLM использует **именно этот** newly-created intent — без extra prompting'а, без few-shot example'а на нём специально.

Это именно та петля, которую центральное утверждение проекта предполагает:

```
work item → LLM extends format → format remembers → LLM uses extension
   ↑                                                          ↓
   └─────────────────── compounding cycle ───────────────────┘
```

Classical Lisp meta-circularity делает это с code = data в одном языке. Здесь meta-circularity делается через format, который **all parties** (human, LLM, host invariants) читают и пишут одинаково.

## Почему metric пишет partial-B

`computeIterationMetrics` measures `compoundingObserved` как:

> tail confirm rate ≥ head + 5pt после confirmed self-mod

После iter 5 head rate = 5/5 = 100%. Tail rate (iter 6-20) = 15/15 = 100%. Diff = 0%, не ≥5%.

Metric **не учёл explicit-name-match** на newly-created intent (это было намеренно отложено в run 1 spec'е, потому что closed enum не позволял такой тест). Run 4 теперь делает name-match measurable, но metric definition осталась с run 1.

**Fix metric** (отдельный SDK patch): добавить `compoundingByNameMatch` — boolean «существует tail-iteration где proposedIntent совпадает с intentId, созданным confirmed propose_meta_intent ранее».
