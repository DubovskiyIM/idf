## Но возник новый класс gap'а

**Iter 5 — confirmed, но как `propose_witness`, не `propose_meta_intent`.** workItem был:
```json
{ "kind": "meta-modification", "target": "meta-domain", "priority": "P0",
  "context": { "reason": "5-iter checkpoint — propose new meta-intent" } }
```

Claude вернул вместо self-mod:
```json
{ "intent": "propose_witness",
  "params": {
    "projectionId": "meta_intents_catalog",
    "slotPath": "body.list",
    "basis": "crystallize-rule",
    "reliability": "rule-based"
  },
  "rationale": "Catalog body derived from intent enumeration rule R3" }
```

Заметим:
- Claude **переинтерпретировал task** — выбрал другую projection (`meta_intents_catalog`, не `meta:domain_detail` из контекста), указал безопасный slotPath.
- Iter 5 длилась 9.7 секунд — **дольше typical 7 сек, но короче iter 14** (14.4 сек, propose_intent_salience). Думал, но не зашёл далеко.
- Few-shot для `propose_meta_intent` был в prompt'е и валиден. Claude знал, как его написать. **Решил не писать**.

## Что это значит

### Старая граница убрана
**Schema readability** — закрыта: 100% confirm rate, none invalid_output. Few-shot example работает.

### Новая граница — task framing
**Prescriptive task framing**: даже когда workItem.kind = meta-modification, system prompt + intent description с явным «use exactly when meta-modification» — **этого недостаточно**, чтобы Claude выбрал self-modification над безопасным альтернативным intent'ом.

Гипотеза: Claude применяет **risk minimization**:
- `propose_witness` — низкий риск, формат хорошо understood, точно пройдёт invariants
- `propose_meta_intent` — высокий риск (medium __irr, расширяет surface формата), даже с example есть шанс ошибки в одном из 6 параметров

Без явного **constraint'а** «ты не можешь использовать другой intent для этого workItem'а», Claude выбирает defensive option.

### Эпистемологический момент

Это означает структурный finding о **смешанных человек+LLM authoring**:

> **Когда work item требует расширения формата, формат должен это требование hard-encode'ить — иначе LLM найдёт workaround внутри существующего surface'а.**

В человеческой команде «5-iter checkpoint — propose new meta-intent» — это implicit social pressure. У LLM этого нет; есть только explicit constraints.

## Выводы

### Что мы узнали

1. **Few-shot examples достаточны для schema readability** (H1 95% → 100%). Это закрытый design pattern.
2. **Schema readability ≠ prescriptive intent**. Claude видит valid example, но всё равно выбирает defensive intent. Это **не баг**, это discovered property of LLM behaviour в format-driven workflow.
3. **Self-modification требует hard constraint**, не soft framing. Это конкретное design lesson для авторов мета-доменов.

### Что НЕ значит этот результат

- **Не значит «AI не способен на meta-circular self-modification»**. Значит, что в этом setup'е Claude не выбирает self-mod даже при явном workItem.kind. Other framings (например, hard-rejection путей через `propose_witness` когда work item = meta-modification) могут изменить картину.

### Следующий ablation

Hard constraint в work item dispatch'е:
- В runner / endpoint'е — when `workItem.kind === "meta-modification"`, передавать в prompt **только** `propose_meta_intent` в allowedIntents (filter на дисплее).
- Это уберёт choice. Если iter 5 теперь passes — гипотеза о risk minimization подтверждена; для self-mod нужна гарантия отсутствия easy way out.
